# 风险与技术债

## 安全问题

### API 密钥泄露
- **严重程度**：高
- **描述**：API 密钥直接硬编码在配置 YAML 文件中（如 `exp4.7/config.yaml` 中的 `api_key` 字段）
- **影响**：密钥可能被提交到 Git 仓库，导致泄露
- **建议**：使用环境变量或 `.env` 文件，将 `.env` 加入 `.gitignore`

### 动态代码执行
- **严重程度**：中
- **描述**：LLM 生成的代码通过 `importlib` 动态执行，无沙箱隔离
- **影响**：LLM 可能生成恶意或危险代码（文件操作、网络请求等）
- **建议**：添加代码白名单校验，限制可用的 Python 模块

## 技术债

### 实验目录膨胀
- **严重程度**：中
- **描述**：存在 6+ 个实验变体目录（exp4.7、exp4.9、exp_4.9_b/c/d/f、4.12exp），每个都复制了类似的核心代码
- **影响**：代码重复严重，修改一处需要同步多处；难以追踪哪个版本是"正确的"
- **建议**：合并到统一框架（`llm_rl_trading_finsaber/` 已在做此事）

### 依赖管理缺失
- **严重程度**：中
- **描述**：没有 `setup.py`、`pyproject.toml` 或统一的 `requirements.txt`（各实验目录各自一份）
- **影响**：环境复现困难，版本冲突风险
- **建议**：创建统一的 `pyproject.toml`

### sys.path hack
- **严重程度**：低
- **描述**：大量使用 `sys.path.insert(0, ...)` 管理导入路径
- **影响**：导入顺序敏感，容易出错，IDE 无法正确解析
- **建议**：重构为标准 Python 包结构

### OpenAI 旧版 SDK
- **严重程度**：低
- **描述**：使用 `openai==0.28`（已废弃的旧版 API）
- **影响**：未来可能不可用，缺少新特性
- **建议**：升级到 `openai>=1.0` 并适配新 API

## 代码质量

### 错误处理不完整
- 多个关键函数缺少异常处理
- LLM 调用失败时缺少重试机制（仅 `lesr_controller.py` 有重试）
- 数据加载失败时缺少优雅降级

### 类型注解不一致
- 部分函数有类型注解，部分没有
- 返回类型标注不完整

### 文档混合语言
- 代码注释和 docstring 主要用英文
- 配置文件和文档有中文注释
- 分析文档（`参考项目梳理/`、`项目分析/`）全中文
- 无统一语言规范

## 性能问题

### 数据加载
- 每次运行重新加载完整 pickle 文件到内存
- 无增量加载或缓存机制
- 大数据集可能导致内存压力

### 并行训练
- `torch.multiprocessing` 并行训练已实现
- 但每个 worker 重复加载数据，未共享内存

## fragile areas（脆弱区域）

- `lesr_controller.py` 中的 LLM 输出解析 — 依赖 LLM 输出格式的稳定性
- `dqn_trainer.py` 中的状态维度计算 — LLM 生成的特征可能改变维度
- `FinMemDataset` 的 pickle 数据格式 — 无版本管理，格式变化会破坏兼容性
