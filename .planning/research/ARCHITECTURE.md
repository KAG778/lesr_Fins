# 架构研究

## 推荐架构：诊断 → 修复 → 验证 三阶段

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  阶段 1: 诊断    │───>│  阶段 2: 修复    │───>│  阶段 3: 验证    │
│                 │    │                 │    │                 │
│ · 现状基准测试   │    │ · Prompt 改进    │    │ · Walk-forward  │
│ · 不稳定性归因   │    │ · 反馈循环重设计  │    │ · 统计检验      │
│ · 特征质量分析   │    │ · 特征筛选机制   │    │ · 多维度验证    │
│ · 问题定位      │    │ · 训练流程重设计  │    │ · 论文级结果    │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

## 组件边界

### 1. 诊断模块（新增）
- **基准测试器** — 多次运行 DQN baseline 和当前 LESR，收集统计样本
- **特征分析器**（增强现有）— 分析 LLM 生成特征的分布、方差、信息比
- **不稳定性归因器** — 分解不稳定性来源（LLM 波动 vs DQN 训练波动 vs 数据噪声）

### 2. LESR 核心改进（修改现有）
- **Prompt 引擎**（改进 `prompts.py`）— 更精确的金融语义引导
- **反馈生成器**（改进 COT）— 基于特征分析结果生成有意义的反馈
- **代码质量守门**（新增）— LLM 输出的语法/维度/数值稳定性检查

### 3. 评估框架（新增）
- **Walk-forward 引擎** — 滚动窗口训练/测试
- **统计检验器** — Bootstrap Sharpe 比率差异、多重比较校正
- **结果聚合器** — 多股票多窗口结果汇总与报告生成

### 4. 训练/验证/测试管道（重新设计）
- 替代固定窗口切分
- 支持多种切分策略（固定、滚动、扩展窗口）
- 防止数据泄露（purged cross-validation）

## 数据流

```
原始 OHLCV 数据
    │
    ├─> 训练/验证/测试切分（Walk-forward 引擎）
    │       │
    │       ▼
    │   训练集 ──> LLM Prompt + 历史反馈 ──> LLM 生成 revise_state()
    │       │                                    │
    │       │                              代码质量守门
    │       │                                    │
    │       ▼                                    ▼
    │   DQN 训练 <─── revise_state() + intrinsic_reward()
    │       │
    │       ▼
    │   特征分析 ──> 反馈生成 ──> 下一轮迭代
    │
    └─> 测试集 ──> 最终评估 ──> 统计检验 ──> 结果聚合
```

## 构建顺序

1. **诊断工具**（优先）— 先理解问题再修复
2. **训练/验证/测试重设计** — 基础设施
3. **LLM 优化循环改进** — 核心修复
4. **全面验证** — 最终确认
