# 技术栈研究

## 当前技术栈评估

| 技术 | 当前使用 | 评估 |
|------|---------|------|
| Python 3.8+ | ✓ | 适配 |
| PyTorch | ✓ | 适配 — DQN/TD3 训练核心 |
| OpenAI API 0.28 | 旧版 | ⚠️ 需升级到 v1.x，但非优先级 |
| SHAP | ✓ | 适配 — 特征重要性分析 |
| FINSABER 回测 | ✓ | 适配 — 继续使用 |
| YAML 配置 | ✓ | 适配 |

## 建议新增

| 技术 | 用途 | 置信度 |
|------|------|--------|
| scipy.stats | 统计显著性检验（t-test, bootstrap） | 高 — 已是依赖 |
| 多重比较校正（Bonferroni/BH） | 多股票多窗口对比时控制假阳性 | 高 — scipy 内置 |
| Walk-forward 分析框架 | 时间序列交叉验证 | 高 — 自行实现 |
| 结果持久化（JSON/CSV） | 结构化存储实验结果用于对比 | 高 |

## 不建议使用

| 技术 | 原因 |
|------|------|
| Stable-Baselines3 | 当前已有 DQN 实现，替换成本高且不解决核心问题 |
| Ray/RLlib | 过重，当前单机训练足够 |
| Optuna/Hyperopt | 超参数调优重要但不是当前瓶颈，先解决 LESR 稳定性 |

## 关键洞察

当前技术栈本身不是问题核心。瓶颈在于：
1. **LLM prompt 设计** — 如何让 LLM 生成更稳定、更有意义的金融特征
2. **评估方法** — 需要统计检验而非单次运行对比
3. **反馈循环** — COT 反馈如何有效引导 LLM 改进
